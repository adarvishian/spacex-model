## What broke tonight in V2.18

Plugin executed §3.1.5 Step 3 + §3.5b + §3.6 + §3.7 + §3.9 + §3.10 against V2.17 (good) but the §3.5b/§3.6 deployment binding triggered a circular dependency:

```
Allocator R88 cash alloc (year T)
  ← R50 sigmoid output (year T-1, prior-year cash lag per spec)
  ← R48 cash demand
  ← Starlink R88 V2 BB sat CapEx (= R33 × R78)
  ← R33 V2 BB launches = MIN(R8 / cost, internal_target)
  ← R8 Starlink V2 BB cash IN
  ← Allocator R88 (loop)
```

Excel's iterative calc converges to **zero** (a stable fixed point) instead of the intended steady state. Group Revenue 2050 collapsed $583B → $17B. R108 = "CHECK" all years.

## Architectural root cause

The Sprint 11 §3.2 spec wrote R48 (sigmoid demand) as `INDEX(Starlink V2 BB sat CapEx, ...)`. This reads the ACTUAL CapEx, which is bound by the cash MIN in §3.6. Circular: sigmoid demand depends on sigmoid output via the cash bind.

**The fix** (deferred to a future sprint — call it Sprint 11f architectural rework): R48 should read the UNCAPPED target demand (anchor × learning × unit cost), NOT the actual cash-bound CapEx. This decouples demand from output:
- R48 sigmoid demand = `$D$33 × (1+learning)^year_offset × E78_unit_cost + facility_capex` (target demand, no cash bind)
- R33 actual launches = `MIN(cash/cost, internal_target)` (cash-bound, uses sigmoid output)
- R88 sat CapEx = R33 × R78 (actual CapEx, downstream of cash MIN)

This breaks the circular: R48 reads anchor+learning, never reads R33 or R8 or R88. Need to apply same pattern to R52 (V2 DTC), R58 (V3 BB), R63 (V3 DTC).
